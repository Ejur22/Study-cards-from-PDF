# 📊 Полное руководство по тестированию StudyCards

## ЧАСТЬ 1: ТЕОРИЯ И КОНЦЕПЦИИ

### 1.1 ЧТО ТАКОЕ ТИПЫ ТЕСТОВ?

#### 🔹 UNIT ТЕСТЫ (Модульные тесты)
**Определение:** Тестирование отдельных функций/методов изолированно

**Пример на backend:**
```python
def test_hash_password():
    """Тест хеширования пароля"""
    plain = "password123"
    hashed = hash_password(plain)
    assert verify_password(plain, hashed) == True
    assert verify_password("wrongpassword", hashed) == False
```

**Пример на frontend:**
```typescript
// Тест функции валидации email
test('isValidEmail должна возвращать true для корректного email', () => {
  expect(isValidEmail('test@example.com')).toBe(true);
  expect(isValidEmail('invalid-email')).toBe(false);
});
```

**Характеристики:**
- ✅ Очень быстрые (миллисекунды)
- ✅ Изолированные (без БД, API, сетевых запросов)
- ✅ Много их может быть (100+)
- ✅ Используют моки и фикстуры
- ❌ Не ловят проблемы интеграции компонентов

---

#### 🔹 ИНТЕГРАЦИОННЫЕ ТЕСТЫ
**Определение:** Тестирование взаимодействия нескольких компонентов

**Пример на backend:**
```python
def test_login_endpoint_with_valid_credentials():
    """Тест полного процесса логина"""
    # Создаём тестового пользователя
    user = create_test_user("test@example.com", "password123")
    
    # Отправляем запрос на /auth/login
    response = client.post("/auth/login", data={
        "username": "test@example.com",
        "password": "password123"
    })
    
    # Проверяем что получили токены
    assert response.status_code == 200
    assert "access_token" in response.cookies
    assert "refresh_token" in response.cookies
```

**Пример на frontend:**
```typescript
// Тест компоненты с реальными API запросами (но мокированными)
test('LoginForm должна отправить данные и показать ошибку при неудаче', async () => {
  const mockLogin = jest.fn().mockRejectedValue({ 
    response: { status: 401, data: { detail: "Invalid credentials" } } 
  });
  
  const { getByText, getByLabelText } = render(
    <LoginForm onLogin={mockLogin} />
  );
  
  fireEvent.change(getByLabelText(/email/i), { target: { value: 'test@example.com' } });
  fireEvent.change(getByLabelText(/password/i), { target: { value: 'wrong' } });
  fireEvent.click(getByText(/войти/i));
  
  await waitFor(() => {
    expect(getByText(/ошибка аутентификации/i)).toBeInTheDocument();
  });
});
```

**Характеристики:**
- ⏱️ Медленнее юнит-тестов (секунды)
- 🔗 Тестируют реальные взаимодействия между компонентами
- 📦 Используют тестовую БД или моки
- ✅ Ловят проблемы на стыках компонентов
- 🎯 20-30% от всех тестов

#### 🔹 СКВОЗНЫЕ ТЕСТЫ (E2E - End-to-End)
**Определение:** Тестирование полного бизнес-сценария через UI, как реальный пользователь

**Пример:**
```typescript
// Cypress E2E тест: пользователь регистрируется, логинится, загружает PDF, проходит тест
describe('E2E: Полный цикл использования приложения', () => {
  it('Новый пользователь регистрируется, загружает PDF и проходит тест', () => {
    // 1. Переходим на главную
    cy.visit('http://localhost:5173');
    
    // 2. Нажимаем "Регистрация"
    cy.contains('Регистрация').click();
    
    // 3. Заполняем форму
    cy.get('input[name="email"]').type('newuser@example.com');
    cy.get('input[name="password"]').type('SecurePass123!');
    cy.get('input[name="full_name"]').type('John Doe');
    cy.contains('Зарегистрироваться').click();
    
    // 4. Перенаправило на главную
    cy.url().should('eq', 'http://localhost:5173/');
    
    // 5. Загружаем PDF
    cy.get('input[type="file"]').selectFile('test.pdf');
    
    // 6. Видим вопросы теста
    cy.get('.quiz-question').should('be.visible');
    
    // 7. Ответляем на 3 вопроса
    for (let i = 0; i < 3; i++) {
      cy.get('.quiz-option').first().click();
      cy.contains('Далее').click();
    }
    
    // 8. Видим результаты
    cy.get('.results-score').should('contain', '%');
  });
});
```

**Характеристики:**
- 🐢 Очень медленные (минуты)
- 🖱️ Тестируют UI, как настоящий пользователь
- ✅ Ловят реальные проблемы пользовательского опыта
- ❌ Нестабильные (могут флейкиться)
- 📊 5-10% от всех тестов (20-50 тестов максимум)

### 1.2 ПИРАМИДА ТЕСТИРОВАНИЯ

```
         ⭐ E2E тесты (5-10%)
        /  \
       /    \
      /      \ Медленные, дорогие
     /________\ 20-50 тестов
    
   🔶 Интеграционные тесты (20-30%)
   /  \
  /    \ Средняя скорость, средняя стоимость
 /      \ 100-300 тестов
/________\

🟢 Unit тесты (60-70%)
_______________ 
         | Быстрые, дешевые
         | 500-2000 тестов
         |
```

**Пирамида объясняет:**
1. **Основание** - много быстрых unit тестов (дешевые в разработке)
2. **Середина** - интеграционные тесты (проверяют взаимодействие)
3. **Вершина** - несколько E2E тестов для критичных сценариев

**Почему именно так:**
- Unit тесты быстро дают feedback разработчику
- Интеграционные тесты ловят проблемы стыков
- E2E тесты проверяют реальные пользовательские сценарии
- **❌ НЕПРАВИЛЬНО:** Много E2E тестов - приложение будет тестироваться часами
- **✅ ПРАВИЛЬНО:** 70% unit, 25% integration, 5% E2E

## ЧАСТЬ 2: АНАЛИЗ ТЕКУЩЕГО MVP

### 2.1 Критические пользовательские сценарии StudyCards

#### Сценарий 1: Регистрация и вход
```
ACTOR: Новый пользователь
STEP 1: Нажимает "Регистрация"
STEP 2: Заполняет email, пароль, имя
STEP 3: Отправляет форму
EXPECTED: Аккаунт создан, автоматический вход
RISK: ❌ ВЫСОКИЙ - аутентификация, валидация, хеширование пароля
```

#### Сценарий 2: Загрузка PDF и создание теста
```
ACTOR: Авторизованный пользователь
STEP 1: Нажимает "Выбрать файл" и выбирает PDF
STEP 2: Система анализирует PDF
STEP 3: AI генерирует вопросы
STEP 4: Пользователь видит тест
EXPECTED: Вопросы загружены, можно проходить тест
RISK: ❌ ОЧЕНЬ ВЫСОКИЙ - работа с файлами, внешний API, хранилище
```

#### Сценарий 3: Прохождение теста
```
ACTOR: Пользователь проходит тест
STEP 1: Видит вопрос с вариантами ответа
STEP 2: Выбирает вариант ответа
STEP 3: Нажимает "Далее" или "Завершить"
STEP 4: Видит результаты
EXPECTED: Все ответы сохранены, результат рассчитан
RISK: ⚠️ СРЕДНИЙ - логика подсчёта, сохранение данных
```

#### Сценарий 4: История обучения
```
ACTOR: Пользователь смотрит историю
STEP 1: Нажимает "История"
STEP 2: Видит список пройденных тестов
STEP 3: Может фильтровать, сортировать, пагинировать
STEP 4: Может скачать PDF
EXPECTED: Все данные загружены, фильтры работают
RISK: ⚠️ СРЕДНИЙ - фильтрация, пагинация, права доступа
```

---

### 2.2 Ключевые бизнес-правила и ограничения

| Правило | Ограничение | Проверить |
|---------|-----------|----------|
| Аутентификация | Пароль ≥8 символов | валидация пароля |
| Файлы | PDF ≤ 10MB | размер файла |
| Роли | User, Admin | проверка ролей |
| Токены | Access: 15мин, Refresh: 7дней | время жизни |
| API | Яндекс.Словарь, PDF обработка | fallback при ошибке |
| Тесты | ≥3 вопроса на PDF | минимум вопросов |

---

### 2.3 Области повышенного риска

```
🔴 ВЫСОКИЙ РИСК:
  ├─ Аутентификация (логин, регистрация, токены)
  ├─ Работа с файлами (загрузка, валидация, хранилище)
  └─ Интеграции с внешним API (Яндекс.Словарь, PDF-to-text)

🟠 СРЕДНИЙ РИСК:
  ├─ Фильтрация и пагинация
  ├─ Ролевые права (User vs Admin)
  └─ Логика подсчёта результатов

🟡 НИЗКИЙ РИСК:
  ├─ Отображение информации
  └─ CSS/оформление
```
